# SocietyOS Architecture & Project State

This document serves as a comprehensive handover guide for any developer or AI assistant joining the SocietyOS project. It outlines the technology stack, core architecture, data models, and current completion state.

## 1. Project Overview & Tech Stack
**SocietyOS** is a multi-tenant SaaS application designed to manage residential societies (HOAs, gated communities). It handles user roles, unit management, maintenance billing, visitor tracking (via QR codes), and helpdesk complaints.

* **Framework:** Next.js 14.2 (App Router)
* **Frontend:** React 18, Tailwind CSS, Radix UI (Shadcn patterns), Lucide React
* **Backend:** Next.js API Routes (Serverless), Node.js, `jsonwebtoken`
* **Database:** MongoDB (via Mongoose 9.7.4)
* **Payments:** Razorpay integration (with webhook idempotency)
* **Testing:** Jest (Integration & E2E tests against live DB)
* **Hosting:** Vercel (Frontend + Serverless Functions) & MongoDB Atlas

## 2. Multi-Tenant Architecture & Security
The most critical engineering pattern in this app is the **Tenant Isolation**.

1. **`AsyncLocalStorage` Context:** When a request arrives, the `withAuth` middleware (`src/lib/auth/middleware.ts`) decodes the JWT and stores the user's `userId`, `role`, and `societyId` in an `AsyncLocalStorage` store (`src/lib/tenant/context.ts`).
2. **`tenantScopingPlugin`:** Every Mongoose model (except SuperAdmin endpoints) is wrapped with `tenantScopingPlugin` (`src/lib/tenant/scopingPlugin.ts`). This plugin intercepts *every* `.find()`, `.findOne()`, `.update()`, etc., and automatically injects `{ societyId: context.societyId }` into the database query.
3. **Security Guarantee:** It is mathematically impossible for an Admin or Resident in Society A to view or modify data in Society B. If a developer forgets to filter by `societyId` in an API route, the plugin handles it automatically. If a background job queries the DB without a context, it throws a fatal error unless explicitly bypassed with `.setOptions({ unscoped: true })`.

## 3. Authentication & Roles
Auth is implemented entirely in-house using JWTs.
* **Access Tokens:** Short-lived, passed in `Authorization: Bearer <token>`.
* **Refresh Tokens:** Long-lived, stored hashed in the DB. Includes "Token Family" tracking to detect token reuse/theft (rotates on every use; revokes all sessions if a used token is presented again).
* **`apiClient`:** The frontend uses `src/lib/api/client.ts` for all requests, which automatically intercepts `401 Unauthorized` responses, hits `/api/auth/refresh`, and transparently retries the failed request.

### Roles
* **Super Admin:** Global system manager. `societyId` is null. Can onboard new societies and activate them.
* **Admin:** Society-level manager. Can add units, approve residents, generate bulk maintenance bills, and assign complaints.
* **Resident:** Tied to a specific `Unit`. Can pay bills via Razorpay, generate time-limited QR codes for visitors, and file helpdesk complaints.
* **Watchman:** Gate security. Uses a mobile-optimized UI to scan QR codes (via `html5-qrcode`) or manually log unscheduled visitors.

## 4. Core Features & Data Models
All schemas are located in `src/models/`.

* **Society & Unit:** The foundational structures. *Note: A society cannot be activated by a Super Admin unless `emergencyContacts` are provided.*
* **User:** Handles auth and profile data. `unitId` is populated for Residents.
* **Visitor:** Tracks entry/exit. Residents can "pre-approve" a visitor, which generates a JWT-signed QR Token valid for a specific time window. Watchmen scan this token to create the `Visitor` entry record.
* **Bill & Receipt:** Generated by Admins. Residents pay via Razorpay. A background webhook (`/api/webhooks/razorpay`) listens for `payment.captured`, updates the Bill to `Paid`, and generates a `Receipt`.
* **Complaint:** Helpdesk ticketing. Enforces a strict status pipeline (`Open` -> `In Progress` -> `Resolved` -> `Closed`). Moving to `Resolved` requires a `resolutionNote`.
* **AuditLog:** Immutable ledger tracking critical actions (e.g., when a complaint status changes).

## 5. Current State & Verification
The project is **100% feature-complete** for the MVP and is ready for deployment to Production on Vercel (pending final manual confirmation).

* **Frontend UI:** All 4 role dashboards (`/superadmin`, `/admin`, `/resident`, `/watchman`) are fully built, styled with Tailwind/glassmorphism, and responsive.
* **API Validation:** All backend routes validate payloads via `zod`.
* **Automated Tests:** The `tests/` directory contains exhaustive Jest integration tests verifying:
  * Strict cross-tenant isolation.
  * Razorpay webhook idempotency (handling duplicate webhook fires safely).
  * QR Code expiry and forgery prevention.
  * Complaint state machine integrity.

## 6. Handover Notes for the Next Developer/AI
* **Database Updates:** If you add a new tenant-scoped collection, you *must* apply `schema.plugin(tenantScopingPlugin)`.
* **Populate Context Loss:** In rare cases, complex Mongoose `.populate()` calls can lose the `AsyncLocalStorage` context. If you encounter `Tenant context not set for find query on [Model]`, pass `.populate({ path: '...', options: { unscoped: true } })`. This is safe because the parent query is already scoped.
* **Env Vars:** The Vercel production environment must have `JWT_SECRET`, `JWT_REFRESH_SECRET`, `CRON_SECRET`, `MONGODB_URI`, `RAZORPAY_KEY_ID`, `RAZORPAY_KEY_SECRET`, and `RAZORPAY_WEBHOOK_SECRET`.
* **Build Exclusions:** `tsconfig.json` excludes the `scratch/` and `scripts/` folders to ensure strict TypeErrors in those files do not break Vercel production builds, but explicitly includes the `tests/` directory to ensure type safety in our testing infrastructure.
